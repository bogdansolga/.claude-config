## 4. Decisions & rationale

- **archiver: stay on v7, do not migrate to v8.** v8 is a breaking ESM rewrite (named-only exports, no callable default), `@types/archiver` hasn't shipped v8 types, and the existing `export.service.ts` code is correct for v7. The actual bug was someone bumping the *runtime* dep to `^8` without updating the import or types ‚Äî reverting the version fixes it with zero code change.
- **`cacheComponents`: fix properly, do not disable.** The user explicitly said "do not disable it, fix it properly." The fix is per-page/per-component Suspense boundaries around dynamic data (auth/cookies, client `usePathname()`), not turning off PPR. Only the dynamic-segment route `[id]` exposed the layout-level un-Suspensed hooks during shell prerender; static-path routes were tolerant.
- **Build on the VPS, don't fight Rosetta.** The deploy script's multi-arch `buildx build` on the Apple-Silicon Mac wedged for ~1 hour on the emulated `linux/amd64` leg (`next build --turbopack` under Rosetta + a stale `multiarch-builder`). Native `docker build` on the x86_64 VPS took ~5 min. The user chose "build on the VPS instead."
- **deploy-diff.sh: single-arch `linux/amd64` by default, arm64 opt-in.** The prod VPS is x86_64; the arm64 leg was redundant work every deploy. arm64 stays available via `DEPLOY_PLATFORMS` for the planned Pi target.
- **deploy-diff.sh: reuse the buildx builder by default (not recreate-each-run).** Recreating wipes the `--mount=type=cache` bun-install cache, ~1‚Äì2 min cold-deps tax per deploy ‚Äî wasteful on the Mac Studio (the user's default machine). Auto-recreate only when the builder is missing/broken; `--fresh-builder` for the rare wedge.
- **Commits via the `/git:commit` skill** ‚Äî user instruction (2026-05-11), saved to memory.
- **APP_SECRET freshly generated** and added to `.env.production` ‚Äî prod had never used it (the running container had no `APP_SECRET`), so nothing was bricked.

## 5. Gotchas / lessons learned

- **`bun verify:all` does NOT run `next build`** ‚Äî only `tsc` + `biome` + vitest. So `next build` breaks (the archiver ESM issue, the PPR Suspense issues) stayed latent. The codebase hadn't been production-built since Jan 2026.
- **`deploy-all-diff.sh` masked failures** ‚Äî parent `set -e` aborted the deploy subshell before `echo $? > statusfile`, so build/sync failures reported as a green "SUCCESS". Fixed in `27c9355` (`set +e` in the subshells) + `sync-diff.ts` now exits non-zero on table failures.
- **`deploy-all-diff.sh --dry-run` only dry-runs the data sync** ‚Äî it still drops `drizzle.__drizzle_migrations` and re-applies all migrations (idempotent here, but real).
- **db-sync leaked the prod DB password** in `Command failed: psql "postgresql://user:PASS@..."` error messages. Fixed in `27c9355` (`redactConnectionStrings`). Also: when constructing `docker run` / `ssh ... <<EOF` with secret interpolation, the heredoc must not be quoted *and* the secrets must have no `$`/backtick (we check before use).
- **`sync-portfolio-data.sh` vs `deploy-diff.sh` had incompatible `REMOTE_HOST` conventions** (one wanted `user@host`, the other a bare host + separate `REMOTE_USER`). Harmonized in `7d89c5c`; now both accept either form.
- **Mac-side multi-arch `next build --turbopack` under Rosetta + a stale `multiarch-builder` = wedge** (~190√ó slowdown, turbopack workers crash-looping). `docker buildx rm <builder>` + recreate fixes a stale builder. The robust answer is to not emulate at all ‚Äî build on the x86 VPS.
- **VPS RAM**: "ubuntu-4gb" but actually ~3 GB, no swap; `next build` peaks ~1.5‚Äì2 GB ‚Üí add a temp 2 GB swapfile before building there (`sudo fallocate -l 2G /swapfile && sudo chmod 600 ... && sudo mkswap ... && sudo swapon ...`; `swapoff` + `rm` after).
- **`git` is not in the Docker build image** (alpine bun/node) ‚Äî `next.config.ts`'s `getGitInfo()` falls back to `dev`/`unknown`; pass `--build-arg BUILD_COMMIT=<sha>` so `/api/health` shows the right commit.
- **psql write statements are sandboxed in this environment** ‚Äî `psql -c "UPDATE/INSERT/DELETE..."` is blocked. Use the app UI or a migration file. (Pre-existing; from the prior handoff.)
- **macOS ships rsync 2.6.9** (2006) ‚Äî `--info=...` and other modern flags fail. `nb install rsync` for 3.x. The exclude-based transfers used this session worked fine on 2.6.9.
